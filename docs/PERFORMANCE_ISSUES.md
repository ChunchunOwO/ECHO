## 详细分析

### 1. `normalizeDuplicateMode` Bug（功能缺陷）

**位置：** `src/main/ipc/libraryIpc.ts`

**问题：**
```typescript
const normalizeDuplicateMode = (value: unknown): DuplicateTrackMode =>
  (value === 'strict' ? 'strict' : 'strict');
```
无论输入什么，永远返回 `'strict'`。`'lenient'` 模式完全失效。

**修复方式：**
```typescript
const normalizeDuplicateMode = (value: unknown): DuplicateTrackMode =>
  (value === 'lenient' ? 'lenient' : 'strict');
```

| 维度 | 评估 |
|------|------|
| 修改风险 | 无 — 纯 bug 修复，一行改动 |
| 安全性 | 无影响 |
| 对用户影响 | 正面 — 恢复 lenient 模式功能 |
| 回归可能性 | 极低 — 逻辑极简 |
| 是否需要测试 | 建议加一个单元测试覆盖两种输入 |

### 2. 远程封面 Worker 无线程池

**位置：** `src/main/library/CoverService.ts`

**问题：**
每次获取远程封面都 `new Worker(...)` 创建一个新线程，启动成本 ~50-100ms。
大曲库（500+ 专辑）触发网络封面补全时，会短时间内创建数百个线程。

**建议修复：**
创建固定大小的 Worker 线程池（4-8 个），复用线程处理封面任务。

| 维度 | 评估 |
|------|------|
| 修改风险 | 低 — 封面获取是独立子系统，不影响播放和库管理 |
| 安全性 | 无影响 |
| 对用户影响 | 正面 — 封面获取更快，系统资源占用更低 |
| 回归可能性 | 低 — 但需确保线程池正确处理错误和超时 |
| 是否需要测试 | 是 — 测试并发获取、单个失败不影响其他任务、池满时排队 |

**注意事项：**
- 线程池需要优雅关闭逻辑（app 退出时 terminate 所有 worker）
- 单个 worker 崩溃不应导致整个池不可用
- 建议先在 dev 环境用 100+ 张专辑测试，观察内存和 CPU 曲线

### 3. SQLite 缺少性能 Pragma

**位置：** `src/main/database/createDatabase.ts`

**问题：**
当前只设置了 `journal_mode = WAL` 和 `busy_timeout = 5000`，缺少几个对大曲库有显著收益的 pragma。

**建议添加：**
```typescript
database.pragma('cache_size = -32768');      // 32MB 页缓存
database.pragma('synchronous = NORMAL');     // WAL 下安全且更快
database.pragma('temp_store = MEMORY');      // 临时表用内存
database.pragma('mmap_size = 268435456');    // 256MB 内存映射
```

| 维度 | 评估 |
|------|------|
| 修改风险 | 低 — 均为 SQLite 官方推荐的 WAL 配合 pragma |
| 安全性 | `synchronous=NORMAL` 在 WAL 下不会丢数据（仅在 OS 崩溃时可能丢最后一个事务，断电安全） |
| 对用户影响 | 正面 — 查询吞吐量提升 20-40%，尤其大曲库受益 |
| 回归可能性 | 极低 — 不改变查询逻辑 |
| 是否需要测试 | 建议在 10000+ 曲目库上对比 benchmark 前后数据 |

**注意事项：**
- `mmap_size` 在 Windows 上支持良好，但建议设上限（256MB）防止虚拟地址空间膨胀
- `synchronous=NORMAL` 的唯一风险：操作系统级崩溃（蓝屏/断电）时可能丢失最近一个 WAL 帧，正常关闭和进程崩溃都不受影响
- 这些改动完全向后兼容，不改变数据库文件格式

### 4. `getStatus()` 热路径对象分配

**位置：** `src/main/audio/AudioSession.ts`

**问题：**
`getStatus()` 每次调用都构造一个包含 ~40 个字段的新对象。该方法被以下路径调用：
- Level meter tick（每 1s）
- Telemetry tick（每 1s）
- 状态转换事件
- 系统音频模式下每 500ms

高频创建大对象会增加 GC 压力，可能导致播放时偶发微卡顿。

**建议修复：**
缓存上一次 status 对象，仅在字段实际变化时重建。或使用 shallow diff 只发送变化的字段。

| 维度 | 评估 |
|------|------|
| 修改风险 | 极低 — 不改变对外接口，仅优化内部分配 |
| 安全性 | 无影响 |
| 对用户影响 | 正面 — 减少播放时 GC 停顿 |
| 回归可能性 | 低 — 但需确保 memoize 逻辑不会返回过期状态 |
| 是否需要测试 | 是 — 验证状态变化时仍能正确推送更新 |

**注意事项：**
- 如果用 shallow compare，注意引用类型字段（数组、嵌套对象）的比较
- 不要过度优化 — 如果实测 GC 停顿不明显，可以推迟此项

### 5. 扫描器缺少目录级 mtime 跳过

**位置：** `src/main/library/TsFileScanner.ts`

**问题：**
增量扫描时，即使目录内容未变化，仍会对每个音频文件调用 `stat()`。
如果目录的 mtime 未变，说明其下文件没有增删，可以直接跳过该目录的 `readdir`。

**建议修复：**
在扫描前记录各目录的 mtime，增量扫描时先检查目录 mtime，未变则跳过。

| 维度 | 评估 |
|------|------|
| 修改风险 | 低 — 仅影响扫描路径，不影响已有数据 |
| 安全性 | 无影响 |
| 对用户影响 | 正面 — 大曲库增量扫描更快 |
| 回归可能性 | 中低 — 某些文件系统（网络驱动器）的目录 mtime 行为可能不一致 |
| 是否需要测试 | 是 — 需测试：新增文件、删除文件、修改文件内容（不改目录 mtime）的场景 |

**注意事项：**
- Windows NTFS 的目录 mtime 在文件内容修改时**不会**更新，只在文件增删时更新
- 这意味着此优化只能跳过"无文件增删"的目录，文件内容变化仍需 stat 检测
- 网络文件系统（SMB/NFS）的 mtime 精度可能不够，建议提供一个"强制全量扫描"选项作为后备
- 建议作为可选优化，默认开启但可在设置中关闭

### 6. IPC 缺少批处理层

**位置：** `src/main/ipc/libraryIpc.ts`

**问题：**
每个库操作（like、unlike、playlist add、track fetch）都是独立的 `ipcRenderer.invoke` 调用。
批量操作（如导入 500 首歌到播放列表）意味着 500 次 IPC 往返。

**建议修复：**
为高频批量操作增加 batch 版本的 IPC handler，一次调用传入数组。

| 维度 | 评估 |
|------|------|
| 修改风险 | **中** — 涉及 IPC 接口变更，preload 和 renderer 都需配合修改 |
| 安全性 | 需确保批量接口同样做输入校验，不引入注入风险 |
| 对用户影响 | 正面 — 批量操作响应更快 |
| 回归可能性 | 中 — 接口变更可能影响现有调用方 |
| 是否需要测试 | 必须 — 需覆盖空数组、超大数组、部分失败场景 |

**注意事项：**
- 这是改动面最大的一项，建议放在最后做
- 可以采用渐进策略：先为最频繁的操作加 batch 版本，保留旧接口兼容
- 不要一次性重构所有 IPC — 逐个迁移，每次迁移后充分测试
- 单个 batch 请求的数组大小应设上限（如 1000），防止主进程阻塞过久

### 7. CoverCacheManager 全量加载

**位置：** `src/main/library/CoverCacheManager.ts`

**问题：**
`updateCoverPathsInDatabase` 一次性 `SELECT * FROM covers` 加载所有行到内存。
万张专辑时产生明显内存尖峰。

**建议修复：**
改为游标式分批处理，每批 500-1000 行。

| 维度 | 评估 |
|------|------|
| 修改风险 | 低 — 仅影响路径迁移逻辑，不影响正常播放 |
| 安全性 | 无影响 |
| 对用户影响 | 正面 — 大曲库不再出现内存尖峰 |
| 回归可能性 | 低 — 逻辑等价，只是分批执行 |
| 是否需要测试 | 是 — 验证迁移后所有路径正确更新 |

**注意事项：**
- 分批时需注意事务边界 — 建议整体仍在一个事务内，只是分批 SELECT
- 如果中途失败需要能回滚

## 推荐执行顺序（从最安全到最需谨慎）

1. **先做 #1** — 一行改动，零风险，立即修复功能缺陷
2. **再做 #3** — 几行 pragma，不改逻辑，立竿见影
3. **然后 #4** — 内部优化，不改接口
4. **接着 #7** — 小范围重构，逻辑等价
5. **然后 #2** — 新增线程池模块，需要完整测试
6. **再做 #5** — 需要考虑文件系统差异
7. **最后 #6** — 改动面最大，建议分多个 PR 渐进式完成

## 通用安全建议

- 每个改动独立一个 commit/PR，不要混在一起
- 改动后跑完整 `npm test`，确认无回归
- 性能改动前后用 `npm run benchmark:library` 对比数据
- 涉及音频路径的改动，务必实际播放测试（不能只靠类型检查）
- 发布前在自己的大曲库（1000+ 首）上跑一轮完整使用流程
- 如果不确定某个改动的影响，可以加 feature flag 先灰度
